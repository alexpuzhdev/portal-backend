# 0003. Канонические модели как контракт интеграции

Дата: 2026-04-29
Статус: принято

## Контекст

К одному и тому же бизнес-понятию (продукт, заявка, контакт) разные
внешние системы предъявляют разные представления: разные имена полей,
разные типы, разные обязательные атрибуты. Если ядро будет работать с
данными «как пришли», то:

- логика ядра окажется завязана на специфику внешних систем;
- замена одного источника на другой потребует переписать половину
  модулей;
- два коннектора одной природы (например, два CRM-источника) будут
  выдавать формально несовместимые данные.

## Решение

Вводится единый **канон** — набор Pydantic-моделей в пакете
`canonical-models` (например `CanonicalProduct`, `CanonicalContact`,
`CanonicalRequest`). Канон — единственный источник истины для ядра:

- Коннекторы транслируют данные внешних систем в канонические модели.
- События на шине публикуются строго в каноническом формате.
- Ядро работает только с каноном; о существовании конкретных систем
  знают только коннекторы.

Канон версионируется: каждое событие несёт `version` в payload; при
несовместимом изменении канона — новая мажорная версия.

## Последствия

### Положительные

- Замена источника данных — это переписывание одного коннектора, ядро не
  меняется.
- Несколько коннекторов одной природы автоматически совместимы между
  собой.
- Единая схема упрощает построение поиска, аналитики, экспорта.

### Отрицательные / компромиссы

- При появлении нового специфичного поля внешней системы приходится
  выбирать: расширить канон или потерять часть данных. Расширения
  канона — нечастая, но дорогая операция (затрагивает все коннекторы и
  ядро).
- Дополнительный шаг трансформации в каждом коннекторе.

## Альтернативы

### Передача «сырых» данных внешних систем

Ядро принимает JSON в исходном формате, кладёт в `JSONB`-поле, разбирает
по месту.

Отказались: размывает бизнес-логику ядра и усложняет тестирование. Любая
аналитика становится ad-hoc парсингом.

### Per-source модели в ядре

Ядро содержит знание про конкретные источники данных
(`Bitrix24Product`, `AmocrmProduct` и т. п.) и работает с ними напрямую.

Отказались: это противоречит принципу слоя-над-системами, ядро превратилось
бы в свалку источников.
