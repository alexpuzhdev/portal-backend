# 0001. Модульный монолит для ядра

Дата: 2026-04-29
Статус: принято

## Контекст

Ядро Portal должно поддерживать множество слабо связанных бизнес-доменов:
ленту, каталог, заявки, документы, кадры, уведомления и т. п. На старте
проекта команда разработки — один человек, операционные ресурсы —
минимальные. Развёртывание и сопровождение распределённой системы
обходится дорого: распределённые транзакции, контракты между сервисами,
наблюдаемость, версионирование API, оркестрация, отдельные пайплайны
деплоя.

При этом домены ядра тесно связаны: пользователь, роль, организация,
аудит — пересекаются в большинстве сценариев. Микросервисная нарезка по
доменам потребует синхронных вызовов между сервисами или
дублирования данных, что при одном разработчике превратится в
бутылочное горлышко.

## Решение

Ядро Portal реализуется как **модульный монолит**:

- Один процесс, один деплой, одна база данных.
- Внутри — модули в `apps/core/app/modules/<name>/`, каждый со своими
  слоями Clean Architecture.
- Логическая изоляция модулей: каждый модуль владеет своими таблицами,
  доступ из других модулей — только через сервисный слой или через
  события.
- Для проверок целостности границ — `import-linter`.

## Последствия

### Положительные

- Минимальные операционные накладные расходы: один deployment, одна
  observability-цель, один CI-конвейер.
- Транзакции внутри ядра — обычные SQL-транзакции без саг.
- Рефакторинг между модулями — обычная правка кода, без миграции
  контрактов.
- При росте проекта любой модуль можно вырезать в отдельный сервис,
  потому что границы уже очерчены логически.

### Отрицательные / компромиссы

- При неаккуратной разработке границы модулей размываются (общая база
  упрощает грязные JOIN'ы). Нужны автоматические проверки.
- Развёртывание любой правки выкатывает всё ядро.
- Один процесс — один потолок по горизонтальному масштабированию;
  отдельные модули нельзя масштабировать независимо.

## Альтернативы

### Микросервисы по доменам

Каждый бизнес-домен — отдельный сервис со своей БД и API.

Отказались, потому что: операционная сложность непропорциональна
текущему объёму задач; распределённые транзакции и наблюдаемость требуют
существенных вложений; команда из одного человека не вытягивает столько
параллельных пайплайнов.

### Монолит без модульной структуры

Отказались, потому что: без явных границ переход на сервисы в будущем
превращается в полный rewrite. Модульная структура — это страховка от
архитектурного долга.
