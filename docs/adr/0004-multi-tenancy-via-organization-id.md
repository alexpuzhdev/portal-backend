# 0004. Multi-tenancy через `organization_id`

Дата: 2026-04-29
Статус: устарело
Заменён: [0009](0009-single-tenant-per-deployment.md)

> **Это решение отменено.** Бизнес-модель проекта изменилась с SaaS на
> self-hosted / managed deployment per tenant. См. [ADR-0009](0009-single-tenant-per-deployment.md)
> для актуального решения. Содержимое ниже сохранено как история.

---

## Контекст

Portal — мультиарендная платформа: одна инсталляция обслуживает множество
организаций. Любая утечка данных между организациями — критический
инцидент безопасности. Вопрос выбора стратегии изоляции арендаторов
является фундаментальным и определяет конструкцию репозиториев,
авторизации, тестов и миграций.

Возможные стратегии:

- Отдельная база на организацию (database-per-tenant).
- Отдельная схема на организацию (schema-per-tenant).
- Общая база и схема, изоляция по `organization_id` (row-level).

Для проекта с ожидаемыми сотнями-тысячами организаций малого бизнеса
первые два варианта дают неподъёмные операционные затраты: миграции
надо проворачивать на каждой схеме, коннекшен-пулы умножаются,
управление учётками БД — отдельная задача.

## Решение

Используется **row-level multi-tenancy через колонку `organization_id`**:

- Каждая таблица бизнес-данных содержит колонку `organization_id`
  (NOT NULL, FK на `organizations.id`).
- Все запросы к БД фильтруются по `organization_id`. Это гарантируется
  на уровне базового класса `OrgScopedRepository` и/или явных параметров
  методов repository.
- `organization_id` берётся из контекста авторизации (JWT-claim),
  никогда — из тела запроса или query-параметра.
- Каждый Repository обязан иметь интеграционный тест на tenant
  isolation: запрос от org A не возвращает данные org B.

Композитные индексы по `(organization_id, ...)` — обязательны для
таблиц с большим объёмом данных.

## Последствия

### Положительные

- Один SQL-движок, одна схема, одни миграции — операционно дешевле.
- Подключение нового арендатора — это одна строка в `organizations`, без
  административных операций над БД.
- Возможны кросс-тенантные аналитические запросы (для self-аналитики
  платформы).

### Отрицательные / компромиссы

- Утечка между арендаторами теоретически возможна при ошибке в коде —
  поэтому фильтрация автоматизирована и проверяется тестами и ревью.
- При очень крупных арендаторах общая база может стать узким местом —
  тогда переход на schema-per-tenant потребует миграции данных.
- Операции вроде «удалить все данные арендатора» — это N команд DELETE
  по таблицам, не один DROP SCHEMA.

## Альтернативы

### Schema-per-tenant

Отказались: миграции по N схемам — операционная боль; библиотечная
поддержка в SQLAlchemy и Alembic есть, но требует дисциплины.

### Database-per-tenant

Отказались: операционная стоимость многократно превышает выгоду на этом
масштабе.

### Использование PostgreSQL Row-Level Security (RLS)

RLS интересен как дополнительный слой защиты, но на старте не
используется: усложняет отладку запросов и тесты, требует явной
установки переменных сессии. Может быть включён позже как defense in
depth — отдельным ADR.
