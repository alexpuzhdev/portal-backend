# 0005. RabbitMQ как шина событий

Дата: 2026-04-29
Статус: принято

## Контекст

В проекте есть несколько процессов, которые должны обмениваться
событиями: ядро, коннекторы, сервис уведомлений (notifier). Между ними
нужен механизм асинхронной связи, который:

- развязывает потребителей и производителей по времени;
- даёт повторные попытки доставки и dead-letter обработку;
- поддерживает несколько потребителей одного потока событий;
- знаком разработчику и имеет качественные клиентские библиотеки на
  Python.

## Решение

В роли шины событий между сервисами используется **RabbitMQ** в режиме
topic-exchange:

- Все события — Pydantic-модели в пакете `event-schemas`.
- Имена событий: `<module>.<entity>.<action>` (например
  `catalog.product.created`).
- Производители публикуют события в topic exchange по routing key,
  совпадающему с именем события.
- Потребители подписываются по шаблону routing key
  (`catalog.product.*`, `connector.#` и т. п.).
- Каждый потребитель должен быть **идемпотентным**: одно и то же событие
  может прийти несколько раз (at-least-once).
- События **не содержат** секреты (токены, пароли) — только идентификаторы.
- Версионирование событий — поле `version` в payload.

## Последствия

### Положительные

- Развязка ядра и коннекторов: коннектор может временно лежать,
  события накапливаются в очереди.
- Из коробки: повторы доставки, dead-letter, observability через
  RabbitMQ management UI.
- Многократно проверенная в продакшене технология.
- Наличие готового SDK (`aio-pika`).

### Отрицательные / компромиссы

- Дополнительный stateful компонент в инфраструктуре (RabbitMQ).
- Нужно следить за идемпотентностью потребителей и схемой dead-letter.
- При крупных нагрузках RabbitMQ требует тюнинга.

## Альтернативы

### Apache Kafka

Лучше масштабируется и хранит историю, но операционно тяжелее и не
оправдан на нашем объёме событий. Возврат к этому вопросу возможен,
если проект вырастет до десятков миллионов событий в день.

### Redis Streams

Уже есть Redis в инфраструктуре — соблазн использовать его и для
событий. Отказались: RabbitMQ даёт лучшую модель доставки и DLX «из
коробки», и один stateful компонент решает одну задачу.

### NATS

Хороший вариант, но команда менее знакома; на старте предпочли
проверенный RabbitMQ.
