# 0007. Clean Architecture как стиль внутри Python-сервисов

Дата: 2026-04-29
Статус: принято

## Контекст

Каждый сервис (ядро, коннекторы, notifier) содержит бизнес-логику разной
сложности. Без явных правил по разделению ответственности код быстро
«срастается»: SQLAlchemy-модели пробираются в роуты, бизнес-правила
размазываются по хендлерам, тестирование требует поднимать всю систему.
Опыт показывает, что переписать такой код потом дороже, чем сразу
заложить структуру.

При этом FastAPI — фреймворк с собственной идиоматикой (роуты как
функции, DI через `Depends`), бороться с которой бессмысленно: цена
будет выше, чем выгода от чистоты архитектуры.

## Решение

Все Python-сервисы (ядро и его модули, коннекторы, notifier) следуют
**Clean Architecture** в адаптации под FastAPI. Слои:

- **Domain** — entities, value objects, domain services, repository
  protocols. Чистый Python; нулевая зависимость от фреймворка, БД,
  HTTP, любой инфраструктуры.
- **Application** — use cases (один use case = один сценарий), DTO
  между слоями, ports (интерфейсы внешних систем). Зависит только от
  domain.
- **Infrastructure** — реализации repository (SQLAlchemy),
  HTTP-клиенты (httpx), event publisher, реализации портов. Зависит от
  application и domain.
- **Presentation** — FastAPI роуты, Pydantic-схемы, DI-зависимости.
  Зависит от всех нижних слоёв.

Правило зависимостей — стрелки только наружу — проверяется автоматически
через `import-linter` (контракты в `apps/<service>/.importlinter`).

Никакой бизнес-логики в роутах: роут принимает запрос, вызывает
UseCase, форматирует ответ. Никакой бизнес-логики в repository: только
доступ к данным.

## Последствия

### Положительные

- Бизнес-логика тестируется без БД и без фреймворка (UseCase + Fake
  Repository).
- Замена технологии в инфраструктуре (например, переход с SQLAlchemy на
  что-то другое) не задевает бизнес-логику.
- Чёткие правила, куда что класть, — каждое решение «где это писать?»
  принимается за секунды.

### Отрицательные / компромиссы

- Накладные расходы на маппинг между слоями (Entity ↔ ORM, DTO ↔
  Pydantic) — больше кода, больше типов.
- Для тривиальных CRUD-сценариев слои выглядят избыточно. Соглашение:
  «избыточная структура с первого дня» лучше, чем «деградация структуры
  через год».

## Альтернативы

### Service-Repository pattern без слоёв

Сервисы хранят бизнес-логику и работают с repository, без явных Entity
и UseCase. Отказались: в крупных проектах сервисы превращаются в
«богов» с десятками методов; нужная структура потерялась бы.

### MVC

Старая идиоматика, плохо ложится на FastAPI и не даёт явного
разграничения domain ↔ infrastructure.

### Active Record

Бизнес-логика прямо в ORM-моделях. Отказались: смешивание persistence и
поведения, плохо тестируется и плохо переносится между источниками.
