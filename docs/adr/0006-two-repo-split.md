# 0006. Разделение проекта на два репозитория: backend и frontend

Дата: 2026-04-29
Статус: принято

## Контекст

Платформа состоит из бэкенда (Python-сервисы) и фронтенда (Next.js
приложение). Решение: один репозиторий с моно-структурой или два
независимых репозитория.

Различия между бэкендом и фронтендом существенны: разные стеки, разные
процессы сборки, разные релизные циклы, разные библиотеки, разный
deploy. Один разработчик одновременно работает либо над одной,
либо над другой стороной — не одновременно.

## Решение

Проект разделяется на **два независимых публичных репозитория**:

- `portal-backend` — все Python-сервисы (ядро, коннекторы, notifier),
  пакеты (`canonical-models`, `connector-sdk`, `event-schemas`),
  инфраструктура.
- `portal-frontend` — Next.js приложение.

Контракт между ними — **OpenAPI**, генерируемый бэкендом. Фронт
генерирует TypeScript-типы из этого OpenAPI и использует их как
single source of truth для API-вызовов.

Локально оба репозитория клонируются рядом в общей директории
`portal-project/`, в которой лежит корневой `CLAUDE.md` со сквозными
правилами.

## Последствия

### Положительные

- Каждый репозиторий имеет свой простой CI и свой простой Dockerfile.
- Независимая публикация: можно открыть любой из репозиториев один
  без оглядки на второй.
- Не нужно учиться монорепным инструментам (Nx, Turborepo) на старте.
- Issues и PR разделены по природе проблемы — это упрощает навигацию.

### Отрицательные / компромиссы

- При изменении контракта приходится трогать два репозитория и держать
  их синхронными (компенсируется генерацией типов из OpenAPI).
- Нет общей CI-проверки, что фронт совместим с бэкендом текущей версии —
  это контролируется вручную и через типы.
- Sharing utilities между бэком и фронтом отсутствует; реализуется
  через копию или через npm/pip пакеты, если потребуется.

## Альтернативы

### Монорепозиторий

Один Git-репозиторий с обоими стэками. Отказались: накладные расходы на
монорепные инструменты и общий CI не оправданы при одном разработчике.
Для команды побольше — можно вернуться к этому вопросу.

### Бэкенд как монорепо без фронта

Был вариант поместить только бэкенд в монорепо (apps + packages — это и
так монорепная структура внутри одного репозитория), а фронт — отдельно.
По факту так и сделано; этот пункт фиксирует его.
