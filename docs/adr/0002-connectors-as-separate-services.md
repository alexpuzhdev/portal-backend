# 0002. Коннекторы и каналы коммуникации — отдельные сервисы

Дата: 2026-04-29
Статус: принято

## Контекст

Portal — слой над существующими системами клиента (CRM, отраслевые
сервисы, учётные системы). Платформа не заменяет эти системы, а
агрегирует данные из них. Подключение к каждой внешней системе требует
**коннектора** — компонента, который читает данные из системы и
транслирует их в каноническую модель Portal.

Коннекторы и каналы коммуникации (email, telegram, push) обладают
характеристиками, отличными от ядра:

- Жизненный цикл независим: появляются, ломаются, обновляются по своему
  расписанию (сменилось API внешней системы → правится коннектор без
  деплоя ядра).
- Природа сбоев другая: внешняя система упала или отвечает медленно —
  коннектор должен изолировать эту проблему, не подвешивая ядро.
- Разные требования к ресурсам: один коннектор может постоянно опрашивать
  сторонний API, другой — слушать веб-хуки.

## Решение

Коннекторы и каналы коммуникации (`notifier`) выносятся в **отдельные
сервисы**:

- Каждый коннектор — отдельный процесс, отдельный Dockerfile, отдельный
  pyproject.toml.
- Связь с ядром — только через шину событий (RabbitMQ).
- Коннекторы используют общий пакет `connector-sdk` (базовые классы,
  утилиты) и работают только с каноническими моделями
  (`canonical-models`).
- Каналы коммуникации (`notifier`) — это потребитель событий ядра; ядро
  не знает, как именно отправляется уведомление.

## Последствия

### Положительные

- Сбой одного коннектора не валит ядро.
- Можно остановить и пересобрать конкретный коннектор без перезапуска
  ядра.
- Чёткая зона ответственности: коннектор отвечает только за интеграцию,
  ядро — за бизнес-логику.
- Добавление нового коннектора не меняет код ядра.

### Отрицательные / компромиссы

- Появляется операционная стоимость: нужно деплоить и мониторить N+1
  процессов.
- Шина событий становится критическим компонентом (см. ADR-0005).
- Контракт между коннектором и ядром — события — должен версионироваться.

## Альтернативы

### Коннекторы как модули внутри монолита

Каждый коннектор — модуль `apps/core/app/modules/connector_<name>`.
Отказались, потому что коннекторы по своей природе разнородны и
нестабильны; сбой во внешнем API при таком подходе мог бы повалить ядро
или привести к утечкам соединений в общем event loop.

### Полностью внешние коннекторы (отдельные репозитории)

Каждый коннектор — самостоятельный проект, разрабатываемый отдельно.
Отказались на старте: ставит порог входа высоким (нужно публиковать
`connector-sdk` как отдельный пакет), команда из одного человека не
вытянет N репозиториев. В будущем — открытая опция (партнёры могут
писать свои коннекторы), но пока живём в едином backend-репозитории.
