# 0009. Single-tenant per deployment с внутренней структурой организаций

Дата: 2026-04-30
Статус: принято
Заменяет: 0004

## Контекст

Изначально (см. [ADR-0004](0004-multi-tenancy-via-organization-id.md))
платформа задумывалась как мультиарендный SaaS: одна инсталляция
обслуживает множество **разных клиентов**, изоляция между ними через
колонку `organization_id` в каждой таблице бизнес-данных. Это было
правильное решение для SaaS-модели, в которой утечка данных между
арендаторами — критический инцидент безопасности.

По итогам разговоров с владельцем продукта **бизнес-модель меняется**:
Portal продаётся как **self-hosted / managed deployment**: один
инстанс = один клиент. Под каждого клиента поднимается свой
docker-compose (или managed deployment), своя БД, свой RabbitMQ, свой
S3-bucket. Никакого общего хостинга разных клиентов в одной БД.

При этом сам клиент часто оказывается не атомарной организацией, а
**холдингом**: группой юрлиц, филиалов, брендов или подразделений.
Типичные примеры:

- Холдинг с несколькими юрлицами (разные ИНН, разные счета, общая
  команда). Бухгалтер обслуживает все юрлица; менеджер по продажам
  закреплён за одним.
- Один юрлицо с географическими филиалами (Москва, Питер,
  Новосибирск). У каждого филиала свой склад, свои клиенты, свои
  сотрудники; перенос между филиалами возможен.
- Несколько торговых марок одного юрлица (бренд-A, бренд-B). Каждый со
  своим публичным storefront, своим каталогом; общая бухгалтерия и
  служба поддержки.

Это означает, что внутри одного деплоя нужна **бизнес-сущность
«организация»** с гибкой структурой и возможностью пользователю
работать в нескольких организациях с разными ролями. Но это —
**бизнес-разделение**, не security-граница: данные одной организации
от другой защищаются ролями (RBAC), а не middleware-фильтром.

## Решение

Принимается комбинация из двух уровней:

### Уровень 1 — между клиентами: physical isolation через деплой

- Один клиент-холдинг = один deployment. Свои Postgres, Redis,
  RabbitMQ, S3, секреты, домен.
- Между клиентами нет общего слоя приложения, поэтому утечка между
  ними физически невозможна.

### Уровень 2 — внутри клиента: организация как бизнес-сущность

- В БД есть таблица `organizations` с **деревом** структурных единиц
  холдинга. У записи есть `parent_organization_id` (NULLable, корень —
  холдинг).
- Каждая организация имеет: name, slug, ИНН/реквизиты (опционально),
  primary brand color, logo URL, флаг `storefront_enabled`. Slug
  уникален в пределах инстанса и используется в URL.
- Связь пользователей с организациями — **многие ко многим** через
  таблицу `user_organization_membership(user_id, organization_id,
  role_id)`. Один пользователь может состоять в нескольких
  организациях с разными ролями.
- Бизнес-сущности (заявки, продукты, склад, новости) имеют колонку
  `organization_id` как обычную FK, **там где это бизнес-релевантно**.
  Для глобальных справочников или общих настроек инстанса
  `organization_id` не нужен.
- Изоляция между организациями внутри инстанса — на уровне **ролей и
  permissions через Casbin**: «менеджер филиала Москва» имеет роль на
  конкретную организацию, видит только её данные. Это бизнес-уровень,
  не security-tenant.
- **Нет** middleware, который обязательно вытаскивает `organization_id`
  из JWT и подкладывает в каждый запрос. **Нет** базового класса
  `OrgScopedRepository`, который требуется наследовать. **Нет**
  обязательных tenant-isolation тестов «запрос от org A не видит
  данные org B» — это контролируется RBAC-тестами и code review, не
  инфраструктурой.
- Кросс-организационная аналитика разрешена и тривиальна: это всё ещё
  один клиент-холдинг, и SQL-агрегаты по нескольким организациям —
  ожидаемый сценарий (отчёты по всему холдингу).

### URL-структура (фронт)

- Портал: `/portal/<orgSlug>/...`. Контекст организации в URL — для
  bookmarking, deeplink в новой вкладке, явности контекста на скрине.
- Storefront: `/<orgSlug>` для любой организации с
  `storefront_enabled = true`. Может быть и у корня-холдинга, и у
  любого узла дерева.
- Логин/регистрация — общие для инстанса; после логина пользователь
  попадает на `/select-org`, если у него несколько membership; иначе
  редирект сразу в `/portal/<orgSlug>/...` его единственной организации.

### Telegram-регистрация (на будущее)

Когда будет реализован вход через Telegram-бот, deep-link несёт
`organization_id` в payload (`t.me/portal_bot?start=org_<id>`); бот
создаёт пользователя и membership к этой организации. Owner
организации генерирует такие ссылки в UI.

### События

- Канон не меняется: события идентифицируют сущности по их id. В
  payload включается `organization_id`, когда событие касается
  бизнес-данных конкретной организации (это полезно потребителям —
  например, notifier роутит уведомления по организациям).

## Последствия

### Положительные

- **Между клиентами физическая изоляция**, никакого риска утечки на
  уровне приложения. Компрометация одного инстанса не задевает другие.
- **Внутри клиента — гибкая бизнес-структура** холдинга, без
  необходимости держать N деплоев под N юрлиц одного клиента.
- **Кросс-организационная аналитика тривиальна** (один деплой, одна
  БД, прямые join'ы по холдингу).
- **Меньше архитектурной обвязки**, чем в чистом multi-tenant SaaS:
  нет middleware-extract из JWT, нет base-class, нет tenant-leak
  тестов как обязательного шага. Многоуровневая изоляция превращается
  в обычный RBAC.
- **Деплой автоматизируется**: один шаблон compose, под клиента —
  только переменные окружения и заполнение профиля корневой
  организации.

### Отрицательные / компромиссы

- **Операционные затраты**: каждый клиент = отдельный deployment. У
  владельца продукта появляется задача «N инсталляций — N апдейтов».
  Решается release-pipeline и автоматизацией; на старте при
  единичных клиентах не критично.
- **Если клиент захочет SaaS-модель** (общий хостинг с другим
  клиентом) — это не вписывается. Возврат к multi-tenant SaaS-стилю
  потребует существенной миграции схемы.
- **Безопасность изоляции между организациями внутри клиента
  становится зависимой от корректности RBAC-настроек**. Утечка
  «менеджер филиала Москва увидел заявки филиала Питер» = баг в
  permissions, не критический инцидент, но ошибка домена. Это
  компенсируется тестами на RBAC и code review.
- **`organization_id` остаётся «вирусной» колонкой в схеме**: у
  большинства бизнес-таблиц он будет, индексы по нему — стандартная
  практика. Не такая чистая схема, как single-tenant без внутренних
  организаций, но это сознательный компромисс ради бизнес-модели.

## Альтернативы

### A. Multi-tenant SaaS (изначальный ADR-0004)

Один деплой обслуживает многих клиентов. Отказались: бизнес-модель
изменилась на self-hosted/managed deployment per tenant. Multi-tenant
обвязка (middleware-extract, OrgScopedRepository, leak-тесты) даёт
постоянный накладной расход без выгоды.

### B. Single-tenant per deployment без внутренних организаций

Один клиент = один деплой, никакой иерархии внутри. Отказались:
типичный клиент — холдинг, моделирование филиалов через одни лишь
роли усложняет бизнес-логику и не покрывает кейсы вроде «у этого
бренда свой storefront».

### C-flat. Плоский список организаций внутри инстанса (без дерева)

Не отказались концептуально — это частный случай выбранного решения
(всё на одном уровне без `parent_organization_id`). Дерево даёт
гибкость на будущее без удорожания на старте.

### D. Schema-per-organization внутри одной БД

Каждая внутренняя организация = своя схема в одной Postgres-инстансе.
Отказались: даёт меньше изоляции, чем отдельный деплой между
клиентами; одновременно усложняет миграции и кросс-организационную
аналитику внутри одного клиента — выигрыша нет.
